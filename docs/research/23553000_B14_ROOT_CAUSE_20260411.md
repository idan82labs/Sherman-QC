## What was checked

### Narrower locality only

Direct `surface_classification` sweeps showed that narrower search radii helped only slightly unless the retry also kept the smaller offset band:

- with `search_radius=26–30` and `min_offset=2.5`, it moved to about `120.45–120.53°`

Evidence:

### `profile_section`

`profile_section` was not the answer:
- it drifted into the `150°+` range on this bend family

So this was not a routing-to-profile fix.

## Actual cause

The existing short-obtuse retry path still missed the winning `B14` combination for two reasons:

1. the retry trigger only activated for very large angle errors (`> 4°`)
2. the trigger depended on `side_balance_score`, but the live `surface_classification` result did not populate that field

In the live path, the imbalance was only visible through:
- `side1_points`
- `side2_points`

For `B14`, that imbalance was severe enough to justify retry:
- approximately `536 / 164`

## Fix

Code:
- `/Users/idant/82Labs/Sherman QC/Full Project-qc-clean/backend/bend_detector.py`

Changes:
- derive side balance from `side1_points` / `side2_points` when `side_balance_score` is absent
- allow the selected short-obtuse retry path to run when:
  - angle gap is still `> 1.5°`
  - and side balance is clearly poor (`< 0.35`)
- include `2.5` in the narrow retry offset candidates

This remains narrow:
- only selected short obtuse bends
- only after a successful but still implausible local result
